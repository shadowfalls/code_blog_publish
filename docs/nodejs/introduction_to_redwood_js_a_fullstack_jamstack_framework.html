Redwoodjs is a fullstack web framework. Redwood features an end-to-end development workflow that combines the best parts of React, GraphQL, Prisma, TypeScript, Jest, and Storybook. Redwood is built by Tom Preston Werner (co-founder of Github). In this tutorial we will explore the different features that Redwood provides.

Using Redwood you can create JAMstack applications very quickly. Both the backend and the frontend code are maintained in the same project. React is used in the frontend, and a node server with GraphQL is used in the backend. Redwood has a predefined project structure

Prerequisites:

  • Redwood requires Node.js version >=14.19 <=16.x
  • We will be going through React, Node.js, GraphQL, and Prisma in this tutorial. Some basic understanding of what they are used for can help. If you want to learn what Prisma is, you can refer this post.

Exploring the folders in RedwoodJS:

The web folder is where the frontend React application is present.

The api folder is where the backend graphQL APIs are present. Lets check the folders inside the api directory.

api/src contains the actually code, and the GraphQL schema.

api/db contains the database schema, and migrations. We will go into more detail on this later in this tutorial.

web/src/layouts contains the layouts that we will be using in our react app. Layouts are like containers that hosts the current component/page

web/src/components is where we can put our react components.

Scaffolding the post modal:

Redwood has a scaffolding tool that will automatically generate the backend service, the graphQL definitions, and the frontend UI for performing a simple CRUD operation based on the post modal. Before scaffold lets understand the folder structure of the backend.

api/db contains the database schema and migrations

api/dist contains the backend build files, we won't need to concentrate on this during development

api/src/directives will contain GraphQL schema directives for controlling access to queries and transforming values.

api/src/functions will contain any lambda functions that your app will need. By default it will have the graphql.js file which is needed for GraphQL API.

api/src/graphql contains your GraphQL schema written in a Schema Definition Language (the files will end in .sdl.js).

api/src/lib contains a few files:

  • auth.js has a few empty functions that can be used as a base to setup our authentication flow
  • db.js has the Prisma client that we can use to communicate with the database
  • logger.js is used for logging

api/src/services is where we will be putting our business logic to manipulate the data

api/types contains the automatically compiled GraphQL types and can be ignored during development

What is a Cell?

In RedwoodJS cells are a simpler and more declarative approach to data fetching. A cell is self-contained performs the fetchs and displays its own. A cell should export several specially named constants that Redwood will understand. Lets have a look at the web/src/components/Post/PostCell Cell to understand what cells are. The PostCell will looks something similar to the following

The QUERY constant contains the graphQL Query

The Loading component will be used when the data is being fetched

The Empty component will be displayed when there is no data

The Failure component is rendered when there is an error

The Success component is rendered when the API call has succeeded

As you can see we have different components for the different states of a component, this makes things easy to reuse the cell in multiple places
